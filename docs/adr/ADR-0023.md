# ADR-0023: Reducer identity at Stage 1: declared version string, hash-shaped

Status: ACCEPTED (owner, 2026-07-08). Implementation review confirmed
the IdentityHash preimage matches the independently derived goldens
(kernel/fold/registry_test.go; sha256sum derivation outside Go).
Recommendation (a) was endorsed at WP-05a plan approval the same day.
Date: 2026-07-08
Register: D23

## Context

WP-05a needs a reducer identity for three consumers: WP-05b snapshot
keys (RFC-0002 §7 — explicitly a "non-normative reminder" — keyed by
"reducer code hash, event schema version, run_id or scope, offset");
plugin.invoked code_hash provenance (RFC-0002 §3, RFC-0004 P5 and §6 —
"code_hash: build-time for (a)"); and verify-mode replay drift
classification (architecture.md §2, Grade 2: hashes-match-but-outputs-
differ = hidden nondeterminism, hashes-differ = version drift,
chain-fails = corruption).

D11 resolved pure-plugin delivery as compile-time registration:
first-party, kernel-trust code rebuilt with the binary. So "the hash
of this reducer's code" has no natural build artifact to point at —
which forces the question of what identity IS at Stage 1.

Why this is an interpretation of RFC-0004 §6, not a contract change:
RFC-0004's change policy freezes only §3 (the purity contract);
Registration is registry machinery — a pre-1.0 CODE contract
(versioning.md §1). No plugin.registered/plugin.invoked event has
been persisted by the runtime (the annotated trace's code_hash
`c0de00…` is an illustrative placeholder), so nothing recorded binds
against the past yet.

## Options

(a) Declared identity: the reducer declares (plugin_id, semver);
    identity_hash = sha256(kernel.Canonical({"plugin_id": <id>,
    "semver": <semver>})). Simple, honest for first-party code,
    manually bumped. Weakness: a forgotten bump degrades drift
    classification (see Decision — loud, not silent).

(b) Build-time content hash of the reducer's code. Automatic, but Go
    cannot honestly scope "this reducer's behavior closure": behavior
    includes the transitive import graph; debug.ReadBuildInfo's
    vcs.revision is repo-granular and dirty-tree-fragile; any
    go:generate scheme hashes either too little (one file) or so much
    it collapses into (c).

(c) Whole-binary hash. Over-broad: every rebuild invalidates every
    reducer's snapshots and reclassifies every unchanged plugin as
    drifted, destroying the Grade-2 signal entirely.

## Decision

RECOMMENDATION: (a). The preimage is pinned verbatim: the canonical
bytes (RFC-0001 §5 rules — sorted keys, NFC) of

    {"plugin_id": <id>, "semver": <semver>}

hashed sha256, lowercase hex. Scope, Handles, and configuration are
NOT identity — config is per-invocation data (RFC-0004 §7); behavior
changes are what semver bumps are for.

Honest weakening, stated: with a content hash, "hash match + outputs
differ ⇒ nondeterminism" is a theorem; with declared identity it is a
discipline-backed inference. A forgotten bump misclassifies version
drift as hidden nondeterminism — but that misclassification is itself
a LOUD, flagged divergence (verify-replay reports it; nothing goes
silently green), and the corpus ratchet's pinned (corpus, reducer)
state hashes (versioning.md §5.2) go red on any behavior change
regardless of bump hygiene. Failure fails loud, not silent. This is
acceptable at Stage 1 precisely because D11 pure plugins are
first-party kernel-trust code.

## Consequences

Locks in: the preimage rule above (it becomes data-contract-adjacent
the moment a real plugin.invoked records it); fold.Registration
carrying ID + Semver; identity_hash as the WP-05b snapshot-key
component — a semver bump invalidates exactly that reducer's
snapshots, which are disposable caches (rebuild is the recovery).

Queues: WP-12 (verify-mode replay) must plant BOTH variants of a
reducer change — with a semver bump (asserted: classified as drift)
and without (asserted: flagged as unexplained divergence) — turning
the known weakness into a standing test.

Revisit triggers, explicit and two:
  (i)  third-party/WASM pure-plugin delivery (RFC-0004 D11 option c),
       where a real artifact hash finally exists and supersedes
       declared identity for that delivery class;
  (ii) the first persisted plugin.registered / plugin.invoked event,
       where identity becomes data-contract-adjacent and the preimage
       rule must be treated as frozen.
