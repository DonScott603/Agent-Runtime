# ADR-0022: Durable append: log layout, torn-tail recovery, durability

Status: PROPOSED (owner rulings at WP-04b plan approval 2026-07-07
accepted the recommendations below, contingent on the stated
requirements; formal acceptance or amendment follows owner review of
the implementation)
Date: 2026-07-07
Register: D22

## Context

WP-04b persists the first real event bytes, so the on-disk shape of
the log must be decided before code lands. RFC-0002 D5: seq is
gapless, assigned on commit, by the single log-writer. RFC-0002 §9.3:
after a crash mid-append, the recovered log must end at an event
boundary with gapless seq. Constitution #4 / ADR-0017 / kernel/log
local law: the log is append-only; stored bytes are never rewritten.
docs/errors.md: SEQ_GAP refuses to open the store and alarms.

Two decisions interact here and are settled together:

1. The container layout is LOCAL LAYOUT, not a data contract — the
   ADR-0021 keystore precedent ("keys/<hex>.json (local layout, not a
   data contract)"). What IS frozen is what goes inside each record:
   the S1/S2/S5 canonical event form (versioning.md §2). But gapless
   GLOBAL seq across per-run chains constrains the layout to one
   serialized append point.
2. Torn-tail policy. A crash mid-append leaves a partial final
   record. "Never rewrite stored bytes" protects COMMITTED events; a
   torn record never committed. What recovery may do with torn bytes
   must be stated precisely, because this text will be cited whenever
   someone asks whether recovery can destroy history.

## Options

Layout:
  (a) Single append-only file `<dir>/events.log` of length+CRC framed
      records. One file = one fd = the D5 single serialized writer
      made physical; one Write + one Sync per commit; recovery is one
      forward scan; the directory-entry durability problem (weakest
      on Windows, see Durability below) is paid once at store
      creation, never per commit. Trade: recovery scans the whole
      file at Open — acceptable at Stage 1 (bodies live in blobs,
      D10), superseded by (b) if it ever hurts.
  (b) Segmented files (000001.seg, ...). Bounds the recovery scan and
      eases compaction, but adds rotation — a new crash surface per
      segment boundary — and pays the directory-durability cost once
      per segment. Premature at Stage 1. The recovery rule below
      (consecutive-from-base) is deliberately segment-ready.
  (c) File-per-event. Rejected: every commit rides directory metadata
      (the weakest Windows durability path), gapless assignment
      becomes a directory-listing race, recovery a full-directory
      sort.

Torn-tail policy:
  (1) WAL-truncate. Recovery scans forward; let B be the end offset
      of the last fully valid record. If [B, EOF) is classifiable as
      one torn append, truncate to B and open.
  (2) Skip-marker: leave torn bytes in place, append a marker, resume
      after it. Rejected: the torn region can be shorter than any
      marker header; forward scans need heuristic resync past
      permanent garbage; the file stops being "valid records
      end-to-end", which is the property everything downstream leans
      on.

## Decision

(a) + (1). Record format (local layout v1; evolve only by new magic —
v1 files stay readable; container migration copies record bodies
verbatim, frozen bytes are never re-serialized):

    file   := magic(8) record*          magic = "ARLOG001" (ASCII)
    record := len(u32 LE) crc(u32 LE) body[len]
    crc    := CRC-32C (Castagnoli, stdlib hash/crc32)
              over len-LE-bytes || body
    body   := kernel.Canonical(sealed event) — the S1/S2 frozen form,
              verbatim; sig stored as-is (zeroing is virtual,
              hashing-only, chain.json _rules)
    maxRecordLen := 1 MiB (may be raised, never lowered)

Canonical bytes as the stored form make byte-identical re-reads
structural: Payload round-trips as json.RawMessage, so recovery
asserts kernel.Canonical(parsed) == body for every record. The CRC
covers len||body so a corrupted length fails the checksum instead of
silently misframing.

COMMIT is defined as: Append's file Sync returned nil. The writer is
mutex-serialized — exactly one Write (the whole record) then one Sync
per append, and append N+1 cannot begin until append N returned.
Therefore every committed record fully precedes, in file-offset
order, all bytes of the at-most-one in-flight append; the only region
that can be torn is a suffix belonging to at most one record.

Why truncation can never destroy a committed event: truncation
removes only bytes after B; every committed record's Sync returned
before any byte beyond it was written; so bytes past B belong to an
append whose Sync never returned — a record that never committed and
was never acknowledged to any caller. "Never rewrite stored bytes"
(constitution #4) protects committed events; a torn record is not
one. Recovery truncation is not an exception to the law — it is the
law's boundary, stated: the log's committed prefix is sacred; bytes
beyond it were never part of the log.

Torn-tail taxonomy. Tail := [B, EOF), B = end of last fully valid
record (B >= 8, the magic). Recovery classifies:

    T1  EOF-B < 8                      torn header
    T2  sane len but B+8+len > EOF     incomplete body
    T3  record frames to exactly EOF   torn content
        but CRC fails
    T4  len = 0 or len > maxRecordLen, garbage header, still within
        and EOF-B <= 8+maxRecordLen    one append's extent
    -> T1-T4: truncate to B, then Sync BEFORE Open completes —
       recovery decisions are themselves durable (owner A3). Open
       succeeds.

    C1  EOF-B > 8+maxRecordLen         cannot be one append
    C2  CRC-valid record failing JSON parse, canonical-byte identity,
        seal re-derivation, chain linkage, or seq — the CRC binds the
        bytes to what the writer wrote, and the writer never writes
        such a record. Includes any invalid record with valid records
        after it (mid-file corruption).
    C3  CRC-invalid record framed short of EOF with trailing bytes —
        ambiguous between a fluke-sane torn header and a corrupted
        committed record.
    -> C1-C3: LOG_CORRUPT, refuse to open, alarm. Refusal paths never
       truncate or write; file bytes are untouched after a refused
       Open. Fail-safe direction: when in doubt, preserve bytes and
       alarm.

Store creation is the one sub-magic case: a file shorter than 8 bytes
whose bytes are a prefix of the magic is a torn creation (Open never
returned; nothing was ever committed) — reset by Truncate(0), rewrite
magic, Sync; never an in-place patch (owner A2). A sub-8-byte file
whose bytes are NOT a magic prefix is LOG_CORRUPT (owner A2).

Honest boundary of the argument: the guarantee holds under the
crash-fault model — fail-stop process/OS/power loss, with the storage
stack honoring fsync. If media silently corrupts a committed final
record, it is indistinguishable from a torn append from file bytes
alone; no single-file design can both detect and preserve at this
layer. Detection of committed-byte corruption is the chain verifier's
domain (CHAIN_BROKEN, never repaired) and anchors (WP-04c) bound the
damage. Symmetrically, recovery ACCEPTS a fully valid final record
whose Sync never returned (crash between write and sync where the OS
flushed anyway, or between sync and return): it is valid,
chain-consistent, and indistinguishable from committed. The
durability guarantee is therefore: at least every acknowledged append
survives. An unacknowledged-but-recovered append is an idempotency
concern upstream (effect payloads carry idempotency keys by design).

Seq base at recovery: recovery enforces CONSECUTIVE seq from whatever
base the first record carries; it does not demand base 1. Gaplessness
(D5) is a writer-assignment discipline — a virgin store assigns 1 and
every assignment is last+1; the file-local check is that property's
projection onto one container, and it is exactly what segments and
compaction (§8) need later. A true gap between valid records is
SEQ_GAP: refuse to open, alarm (docs/errors.md — "should be
impossible (D5 gapless); treat as corruption").

Errors: refusal cases C1-C3 carry a new code LOG_CORRUPT. This ADR
authorizes the errors.md row:

    LOG_CORRUPT | log recovery | refuses to open store + alarm | none
    | framing/CRC/content corruption of committed bytes; torn tails
    (ADR-0022) are truncated, never corruption

LOG_CORRUPT is never recorded inside events (not S4-frozen), but
errors.md law forbids ad-hoc error strings, so the code enters the
table with this ADR. Chain-linkage refusals wrap *ChainBrokenError
(errors.As-able) to keep the CHAIN_BROKEN vocabulary without claiming
the verifier-reducer row. After any failed append the store is
poisoned until reopened (the tail may be torn); poisoned calls return
a distinct sentinel wrapping the original failure.

Per-platform durability (stated, not implied-POSIX):

  - os.File.Sync(): Windows -> FlushFileBuffers (written data + file
    metadata, with a disk cache flush); Linux -> fsync (data + inode
    metadata). Per-FILE guarantees on both.
  - File CREATION durability is a DIRECTORY property: POSIX requires
    parent-directory fsync for the entry to be durable; Windows has
    no supported user-mode directory flush reachable from Go — it is
    best-effort and its failure is ignored (the exact vault/ADR-0021
    precedent, same comment language).
  - The single-file layout pays this once: Open creates events.log,
    writes the magic, Syncs the FILE, then best-effort Syncs the
    directory. A crash that loses the just-created file loses an
    empty log — zero committed events, since no Append can have
    returned before Open did. Every subsequent commit is an append to
    an existing directory entry: file Sync suffices on both
    platforms. NTFS metadata journaling additionally makes creation
    durable in practice.

## Consequences

Locks in: record framing v1 and the single-file events.log layout
(local, migratable by verbatim record-body copy); the LOG_CORRUPT
row; truncate-only-torn-tails recovery with durable (Synced)
truncation; refuse-don't-repair for everything that is not a torn
tail; store poisoning after a failed append until reopen.

REQUIREMENT on WP-04c (owner A1, condition of the any-base ruling):
the anchor attests, per container, the base seq and first event_id
alongside run heads. File-local consecutiveness-from-any-base is
accepted ONLY because anchoring closes front-truncation — an attacker
who deletes a valid prefix of the file must break the anchor, not
just the local checks.

Unlocks: WP-05a fold reads over recovered stores; WP-04c anchors over
recovered run heads. Queues: segmentation/compaction container
(§8), inter-process locking (daemon mode), an iterator read API.
