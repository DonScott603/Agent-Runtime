# ADR-0021: Vault-lite blob format and key hierarchy

Status: ACCEPTED (owner, WP-03 plan approval 2026-07-07; human review
of vault/ completed 2026-07-07, no amendments). The whole-file-address
and golden-vector decisions were accepted explicitly at plan approval;
the remaining sections were accepted subject to the required human
code review of vault/ (security-critical path, CLAUDE.md) and stand
as written.
Date: 2026-07-07
Register: D21

## Context

WP-03 implements RFC-0002 §6 (blobs) and D9/D10 (ADR-0009/0010).
Writing the first blob fixes bytes that log envelopes reference by hash
forever, so the format, cipher, and key hierarchy must be pinned before
code lands. Constraints: no new dependencies in a security-critical
path without an ADR; the log references hash-of-ciphertext immutably,
so stored ciphertext can never be re-encrypted; ADR-0009 requires
per-run keys wrapped by a master key with "individual-blob erasure
re-wraps the run key excluding the target".

## Options

Cipher:
  (a) AES-256-GCM from the Go standard library — no new dependency.
  (b) XChaCha20-Poly1305 — larger nonces, but requires golang.org/x/crypto.
Key hierarchy (reading of D9):
  (c) Blobs encrypted directly with the run key; per-blob erasure
      re-encrypts survivors — REJECTED: changes ciphertext, changes
      addresses, breaks immutable log references.
  (d) HKDF-derived per-blob keys — rejected: two Get code paths, and
      the keystore stops being the run→blobs index that rotation needs.
  (e) Random DEK per blob, wrapped by a per-run KEK, wrapped by the
      master key; per-blob erasure rotates the KEK and re-wraps
      survivors excluding the target. The only coherent reading of D9
      under immutable addresses.
Blob address:
  (f) SHA-256 over the entire stored file (header + nonce + ct + tag).
  (g) SHA-256 over the GCM output only — most literal "hash of
      ciphertext", but leaves header/nonce outside the address and
      needs offset arithmetic.

## Decision

(a) + (e) + (f). Owner accepted (f) explicitly at plan approval, plus
the decision to pin the format with golden vectors
(docs/vectors/blob.json), whose expected values are computed by an
independent implementation, never by the Go code under test.

Format v1 (frozen; evolve only by new version byte = new files with
new addresses; v1 files stay readable forever):

    blob file  = 0x01 || nonce(12) || AES-256-GCM ct||tag
                 key = per-blob DEK (32 bytes), AAD = the 1-byte header
    address    = SHA-256 over the ENTIRE file, lowercase hex
    wrapped key = nonce(12) || AES-256-GCM ct(32)||tag(16)
                 run-key wrap: KEK = master key, AAD = raw run id bytes
                 DEK wrap:     KEK = run key, AAD = 64-char hex address
    keystore   = keys/<hex(run_id)>.json (local layout, not a data
                 contract): {version, run_id, wrapped_run_key,
                 blobs: {address: wrapped DEK}}; plain encoding/json
                 (mutable, never hashed, never log-bound)
    master.key = 32 raw bytes, file-based at Stage 1

Entropy is injected raw into the store (crypto/rand.Reader at the
composition root), never via the recorded kernel.Entropy handle: that
handle's answers are logged as rng.seed events, which would write key
material into the log. A refactor toward kernel handles here would be
a key-material-in-log incident, not a consistency improvement.

Erasure semantics: erase-run = destroy the run keystore; erase-blob =
rotate the run KEK, re-wrapping survivors and excluding the target;
blob files are never touched (references stay intact). Erased content
IS distinguishable from never-stored at store level (ErrErased = blob
file present, no key; ErrNotFound = no blob file). ORPHANED blobs — a
crash between blob write and keystore save — are indistinguishable
from erased; they are harmless and never garbage-collected. Erasure
operations are idempotent. Because of that idempotency, a nil return
from erase-blob means the requested (run, address) pair holds no key
afterward — NOT that a key was destroyed by this call; the WP-04
redaction flow must confirm the key existed under that run before
appending a signed redaction.applied, or it can record an erasure
that did not happen. The audit-grade distinction between erasure and
tampering is that signed redaction.applied event (RFC-0002 §5), which
lands with the log in a later work package.

## Consequences

Locks in: AES-256-GCM v1 blob bytes and whole-file addressing (hash
values land in S2-frozen envelopes); no dedup by design (random
DEK/nonce means identical plaintext yields unrelated blobs — this is
what keeps per-run erasure isolation and avoids a plaintext-equality
oracle); stdlib-only crypto, no go.mod changes.

Stage-1 caveats (accepted, addressed by the Stage-2 platform
keystore): key "destruction" is file replacement/unlink without media
scrubbing; KEK rotation defends only against retained old keystore
copies whose bytes are actually gone; no master-key rotation;
best-effort memory zeroing only. The keystore is as durability-
critical as the log — its corruption is equivalent to erasure of the
entire run — so keystore and blob writes fsync before an atomic
rename (best-effort parent-directory fsync after), and EraseBlob's
rotation rewrite is the highest-exposure path and the first place to
look on any report of unreadable content.
