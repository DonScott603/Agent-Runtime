# ADR-0024: Anchor payload schema and Merkle construction

Status: PROPOSED (recommendations endorsed at WP-04c plan approval
2026-07-08 — owner rulings on alarm identity, empty-store refusal,
export surface, plus riders R1/R2 folded below; formal acceptance at
implementation review per the ADR-0022/0023 pattern)
Date: 2026-07-08
Register: D24

## Context

WP-04c writes the anchor event of RFC-0002 D4: a Merkle root over all
current run heads, appended periodically so that whole-run deletion
and container front-truncation become detectable. ADR-0022
(Consequences, owner A1) hardened this into a requirement: recovery's
consecutive-from-any-base rule was accepted ONLY because the anchor
attests, per container, the base seq and first event_id — deleting a
valid file prefix must break the anchor, not merely local checks.

`anchor.appended` is the first NEW event type beyond the RFC-0002 §3
initial taxonomy. §3 is an open set (new types additive, M2), so no
RFC edit is needed — this ADR is where the type is documented. Its
payload schema becomes data-contract at first persistence (ADR-0017;
versioning.md S4 shipped payload schemas, S5 chain-and-anchoring
rules), so every construction parameter below is frozen the moment
one anchor is committed, and each is pinned by docs/vectors/
anchor.json under the golden-file law.

## Options

Merkle construction:
  (a) RFC 6962 MTH verbatim: leaf/interior domain separation
      (0x00/0x01), split at k = largest power of two < n, no node
      duplication, MTH({}) = sha256 of the empty string. A public,
      widely implemented spec: independent derivations for the
      goldens exist off the shelf (process.md §5), and audit-path /
      inclusion-proof shapes are already defined for later — the
      construction PERMITS proofs without freezing anything new.
  (b) Pair-and-promote with odd-node duplication (Bitcoin-style).
      Rejected: duplication admits the CVE-2012-2459 mutation class
      (two leaf sets, one root), and it is a private dialect.
  (c) Flat hash over the concatenated sorted heads (no tree).
      Rejected: verification is all-or-nothing — no inclusion proofs
      are ever possible, which forecloses the D4 export/audit story
      the anchor exists to serve.

Payload contents:
  (i)  Root only. Smallest, but the root is then recomputable only by
       a holder of the full log — a future signed, exported anchor
       could not be checked off-machine against individual runs.
  (ii) Root + the run->head map + the container attestation. The root
       recomputes from the payload alone. D10 (bodies-in-blobs) does
       not apply: heads are integrity metadata — hashes and run
       identifiers, no personal content (threat-model.md places no
       constraint on run_ids; D10's own rationale — chain
       verification never touches personal content — argues FOR
       inline heads). Size is bounded by distinct runs, not events.

Anchor-mismatch alarm identity:
  (1) New errors.md code ANCHOR_MISMATCH, row authorized by this ADR
      (the LOG_CORRUPT/ADR-0022 precedent).
  (2) Reuse CHAIN_BROKEN with new detail terms. Rejected:
      CHAIN_BROKEN's recorded contract is report-once-then-freeze-
      the-run; anchor alarms freeze nothing — one code with two run
      behaviors would be dishonest in errors.md, and consistency
      properties would have to filter on detail strings, not codes.

Empty store:
  (A) WriteAnchor refuses (plain error, never poisoning). An anchor
      attests committed history; an empty container has none, and the
      base attestation would be self-referential — a payload carrying
      the anchor's OWN event_id is a sha256 fixpoint, computationally
      infeasible.
  (B) Write an empty-set anchor with a null/self base convention.
      Rejected: complicates the frozen schema for a case with nothing
      to attest.

## Decision

(a) + (ii) + (1) + (A). Pinned exactly (anchor.json _rules are the
normative statement; this section is its rationale):

Merkle construction — RFC 6962 MTH over the head set:
  - leaf preimage = 0x00 || kernel.Canonical({"head": <lowercase hex
    head>, "run_id": <run id>}); leaf hash = sha256(preimage).
    Canonical (RFC-0001 §5) makes NFC normalization and escaping of
    run_id identical to the stored envelope form for free.
  - Leaf order: ascending NFC-normalized run_id in UTF-16 code-unit
    order — the ADR-0020 rule, so the project has ONE ordering law;
    the leaf sequence equals the key order of the canonically
    serialized heads object. The implementation DERIVES the order
    from Canonical's emitted key order rather than re-implementing
    the comparator, so divergence from canon is impossible by
    construction (the fold.go ethic); a property test additionally
    welds the full construction to an independent reference, and the
    astral-ordering golden pins it against independent derivations.
    Canon's refusals (invalid UTF-8, NFC key collision) are thereby
    AnchorRoot's refusals. Input is a map: duplicate run_ids are
    unrepresentable.
  - Interior node = sha256(0x01 || left || right) over the RAW
    32-byte child digests. Domain separation 0x00/0x01 is the
    second-preimage defense and what inclusion proofs later require.
  - Subtrees split at k = the largest power of two < n; no
    duplication.
  - Empty set: root = sha256 of the empty string. Single leaf: the
    root IS that leaf's hash (sha256 of the 0x00-prefixed preimage).
    Lowercase hex rendering.
  - AnchorRoot({}) is total and vector-pinned even though the store
    never writes an empty anchor: the pure function's empty case is
    law; the store's refusal is policy on top of it.

Anchor payload (canonical bytes, RFC-0001 §5; no floats; no
omitempty — all keys always present):

    {
      "container": { "base_seq": <u64>, "first_event_id": "<hex>" },
      "heads":     { "<run_id>": "<head hex>", ... },
      "merkle_root": "<hex>"
    }

  - container is the ADR-0022 A1 attestation, named with the ADR's
    "per container" language. Stage 1: one events.log = one
    container.
  - heads is the full snapshot at write time, anchor excluded. The
    "" entry IS RFC-0002 D4's "latest owner-scope event", realized as
    a map entry, not a separate field (the "" chain's head is by
    construction the latest owner-scope event id at snapshot time; if
    no owner-scope event exists yet, "" is absent). Because the
    anchor itself is appended to the "" chain, each anchor's heads
    include its predecessor anchor: anchors chain automatically.
  - merkle_root must recompute from heads by the rules above.

Anchor envelope: run_id "" (owner scope — the anchor gains chain
tamper-evidence via the "" chain before its content is ever parsed),
principal "service:kernel", type "anchor.appended", type_version 1,
blobs [], sig null. UNSIGNED AT STAGE 1 IS A DECISION, not an
omission: owner-key signing is Stage-2 vault work (roadmap H4); the
payload is designed so signing and custody are H4's only work — the
export format needs no design, it is the already-frozen canonical
event form. WriteAnchor's sealed return is the Stage-1 export
surface. ts/mono are caller-supplied like every append (the writer
never reads time).

Container identity at Stage 1: the STORE knows base as its first
record (recovery scan, or the first append on a virgin store). The
FOLD-side verifier knows base as the first event it ever folds,
recorded AS-OBSERVED (the stored event_id, before any checks — owner
R1). The two are equivalent exactly when the fold consumed the
container from its start: true for every Stage-1 fold; a
snapshot-resumed fold carries base inside its state; a FILTERED read
would observe a false base — filtered reads do not exist at Stage 1
and are a revisit trigger below. Consequence of as-observed recording
(R1): a tampered first event yields both CHAIN_BROKEN at seq 1 and
anchor_base against an honest anchor — the freeze-nothing converse
extended to the base attestation; designed, the container is already
alarmed.

Verifier semantics (plugins/chainverify 0.2.0 — the first real
ADR-0023 semver bump):
  - Anchor alarms FREEZE NOTHING. An anchor is a claim about the
    past, not a gate on the future: a false claim does not invalidate
    the per-run chains, each of which remains independently
    verifiable, so the verifier alarms (one alarm per failed check)
    and keeps threading heads — including the anchor event itself.
    Freezing on an anchor mismatch would destroy ongoing
    tamper-evidence on the strength of one bad claim. Broken is
    untouched by anchor alarms. Converse, stated: a run broken
    EARLIER freezes the verifier's head while the store's advances,
    so an honest later anchor also alarms — designed behavior (the
    container is already alarmed), not a false positive.
  - Content checks run against the verifier's OWN state: recompute
    the root from its own held heads; compare the payload's container
    attestation to its own observed base. payload.heads is NOT the
    comparison source — it exists for off-machine auditors; the
    own-heads recomputation subsumes it on the fold path.
  - Malformed-payload boundary: a schema-required gate runs FIRST —
    unmarshal failure, nil or empty heads, empty merkle_root, zero
    container fields, or root-recomputation failure alarm as
    anchor_payload; only well-formed payloads reach the root/base
    comparisons (so {"heads":null} can never masquerade as
    anchor_root).
  - Uniform check: every anchor.appended event is content-checked
    regardless of the run_id it rides on. The kernel only writes
    anchors at ""; a hand-placed anchor inside a run is checked
    identically — no special-casing.
  - Corollary of (A), tested: an anchor.appended appearing as a
    container's FIRST event is always hostile or hand-crafted and
    deterministically alarms anchor_base (its base claim would need
    its own event_id — the sha256 fixpoint).
  - Version gate: chainverify declares Handles = {"anchor.appended":
    1} — the first version gate on the integrity watchdog. A future
    v2 anchor without an upcaster makes the WHOLE view
    sticky-unavailable (SCHEMA_UNKNOWN_VERSION), consciously
    accepted: gating yields honest unavailability where gatelessness
    would yield false anchor_payload alarms from misparsed v2
    payloads — a false alarm from the integrity watchdog is the worse
    failure (owner R2).
  - Alarm shape: Code ANCHOR_MISMATCH; fixed detail vocabulary
    anchor_payload | anchor_root | anchor_base; structured
    expected/got fields only (A1 discipline — never error text).

Errors: this ADR authorizes the errors.md row:

    ANCHOR_MISMATCH | verifier reducer | integrity alarm (owner
    notification) | none (read path) | impeaches the anchor, not the
    chain — linkage evidence stands; heads keep advancing; never
    repaired

Record-size honesty: at roughly 80 bytes per head, maxRecordLen
(1 MiB, ADR-0022) bounds an anchor to ≈13,000 heads; store heads
never evict, so the bound is on LIFETIME-distinct run_ids in one
container, not concurrent runs. Past the bound WriteAnchor fails
CLEANLY (encodeRecord errors before any byte is written; the store is
not poisoned). Relief valve when it ever binds: segmentation/
compaction (RFC-0002 §8) or a new type_version — nothing frozen here
prevents either.

## Consequences

Locks in (at first persisted anchor): the payload schema above (S4)
and every Merkle construction parameter (S5), pinned by
docs/vectors/anchor.json; the anchor.appended envelope conventions;
the ANCHOR_MISMATCH errors.md row; the freeze-nothing alarm
semantics; chainverify 0.2.0 identity.

Satisfies: ADR-0022 owner A1 (per-container base attestation closes
front-truncation); RFC-0002 D4's anchor event at Stage 1 (unsigned,
manual cadence — signing is H4, scheduling is explicitly out of
WP-04c scope).

Unlocks: signed off-machine export (H4) with zero format work; Merkle
inclusion proofs later without re-freezing (domain separation is
already in place).

Queues / revisit triggers:
  (i)   Stage-2 signing: the anchor gains sig by the owner key; no
        payload change.
  (ii)  Filtered or offset reads (segmentation era): the fold-side
        base equivalence in Container Identity must be revisited
        alongside RFC-0002 §8 work.
  (iii) Anchor size at the maxRecordLen bound: shard or blob via a
        new type_version.
